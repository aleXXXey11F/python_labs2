# Лабораторная работа №1: Класс и инкапсуляция

## Тема: Транспорт (класс Bus)

### Мыслительный процесс и ответы на вопросы

#### 1. Какие атрибуты были выбраны и почему?

| Атрибут | Тип | Почему выбран |
|---------|-----|---------------|
| `_route_number` | str | Идентифицирует маршрут, обязательный атрибут |
| `_capacity` | int | Определяет максимальную вместимость, есть ограничения |
| `_average_speed` | float | Важно для расчетов времени, есть разумные пределы |
| `_driver_name` | str | Может отсутствовать, но требует валидации |
| `_current_passengers` | int | Отражает текущее состояние, изменяется в процессе работы |
| `_is_on_route` | bool | Определяет логическое состояние объекта |

**Почему именно эти атрибуты?**
- Они покрывают все типы данных (str, int, float, bool)
- Требуют разных видов валидации
- Позволяют демонстрировать изменение состояния
- Есть вычисляемые свойства (`free_seats`)

#### 2. Почему атрибуты сделаны закрытыми?

Атрибуты сделаны закрытыми (с префиксом `_`) по нескольким причинам:

1. **Инкапсуляция**: Скрываем внутреннюю реализацию от внешнего мира
2. **Контроль доступа**: Предоставляем доступ только через свойства (@property)
3. **Валидация**: При изменении значений через сеттеры мы можем проверить корректность
4. **Согласованность**: Предотвращаем прямое изменение атрибутов, которое могло бы нарушить логику работы

Например, прямое изменение `_current_passengers` могло бы привести к превышению вместимости, поэтому доступ к нему только через методы `board_passenger()` и `alight_passenger()`.

#### 3. Какие магические методы реализованы и для чего?

| Метод | Назначение |
|-------|------------|
| `__init__` | Конструктор, инициализация объекта с валидацией |
| `__str__` | Для пользователей: красивое представление с эмодзи и форматированием |
| `__repr__` | Для разработчиков: полная информация для воссоздания объекта |
| `__eq__` | Сравнение автобусов по номеру маршрута и вместимости |

**Почему именно такое сравнение в `__eq__`?**
Автобусы считаются одинаковыми, если у них совпадают основные характеристики (маршрут и вместимость), даже если разные водители. Это логично для транспортного предприятия.

#### 4. Какие бизнес-методы реализованы и какую логику они реализуют?

| Метод | Логика |
|-------|--------|
| `board_passenger()` | Посадка пассажира с проверкой наличия мест и статуса маршрута |
| `alight_passenger()` | Высадка пассажира с проверкой наличия пассажиров |
| `start_route()` | Отправление на маршрут (проверка наличия водителя) |
| `end_route()` | Завершение маршрута (сброс пассажиров) |
| `calculate_travel_time()` | Расчет времени в пути на основе скорости |
| `get_efficiency_rating()` | Оценка загрузки, поведение зависит от состояния |

**Особенности логики:**
- Нельзя посадить пассажира, если автобус не на маршруте
- Нельзя отправить автобус без водителя
- При завершении маршрута все пассажиры выходят
- Рейтинг эффективности меняется в зависимости от заполненности
